# HalfLife Caffeine v1.1.0 — Design

**Date:** 2026-04-24
**Target release:** v1.1.0 (first update after the 2026-04-24 store launch)
**Prior spec:** `2026-04-16-halflife-caffeine-design.md` (v1.0)

## Overview

Three independent workstreams shipping in one release:

1. **Device expansion** — 28 new device IDs added to `manifest.xml`. Mechanical, no code changes.
2. **Companion settings page live** — host the existing `companion/settings/index.html` on GitHub Pages and register the URL with the Connect IQ developer portal so the custom settings page (including preset editor) appears in Connect IQ Mobile. Fix the page to load current values from the watch on open.
3. **Watch history view** — new `HistoryView` joining the existing swipe chain. 14-bar daily chart + scrollable day list. SELECT drills into a past day. Daily totals persisted separately for 90 days.

Risk ordering, lowest to highest: device expansion < companion hosting < watch history. Build + test in that order so blocked history-view work can never hold up device-list or companion readiness.

## Out of scope (deferred to v1.2.0)

- Companion History and Trends tabs going live with real synced data. The tabs exist in the HTML but aren't advertised or relied on in v1.1.0.
- Any features from the "brainstorm other handy features" bucket.
- Base fenix 5 / 5s / 5x support (would require dropping `minApiLevel` to 3.1.0, MIP palette work, 64 KB memory audit, 218×218 layout for fenix5s).
- fenix 8 Pro 51 mm (not in SDK at time of writing).

## Workstream 2 — Companion hosting

### Problem

The repo contains `companion/settings/index.html` — a working custom settings page with preset editor, history, and trends tabs. It is not registered with the Connect IQ developer portal, so Connect IQ Mobile only shows the native typed settings from `resources/settings.xml` (daily limit, bedtime H/M, 3 alert toggles). Users cannot edit drink presets from the phone.

### Solution

1. **Move and host the page on GitHub Pages.**
   - Move `companion/settings/index.html` → `docs/settings/index.html`. Single source of truth, served directly from the `main` branch via GitHub Pages.
   - Enable Pages in the GitHub repo settings (source: `main` branch, `/docs` folder).
   - Resulting URL: `https://jame581.github.io/HalfLifeCaffeine/settings/`.
   - The `companion/` directory becomes obsolete; delete it in the same commit that introduces `docs/settings/`.

2. **Register the Pages URL with the Connect IQ developer portal.**
   - Navigate to the app listing, set the "Settings URL" field to the Pages URL.
   - Metadata-only change; does not require re-uploading the `.iq` bundle but does require resubmission of the store listing for review.

3. **Code fix: load current values on page open.**
   - Connect IQ Mobile opens the custom settings URL with current property values passed in the URL. The exact mechanism (query string vs. hash fragment, parameter name, encoding format) must be verified against Garmin's current custom-settings-page documentation at implementation time — Garmin has changed this between SDK generations.
   - On page load, parse the URL, populate form fields (`dailyLimit`, `bedtimeHour`, `bedtimeMinute`, `alertLimitWarning`, `alertLimitReached`, `alertSafeToSleep`, `presets`).
   - Fall back to current hardcoded defaults if the payload is absent or unparseable (safe for local file:// testing).

4. **Verify end-to-end preset sync.**
   - With the companion page pointed at the GitHub Pages URL, open in Connect IQ Mobile, edit a preset, save.
   - Confirm the watch's `DrinkPresets` reflects the edit after the next `onPhoneMessage` handler run.
   - No code changes expected — `SyncManager.handlePhoneMessage` already handles the `presets` key — but testing against a real device is part of the v1.1.0 verification because this path has never been exercised with a live hosted page.

### Out of scope in this workstream

- History and Trends tabs remaining in the HTML but not wired up to reliable data. They work on best-effort local `localStorage` data (as written today) and are not advertised in the v1.1.0 release notes.
- Page restyling / branding changes. The current dark-teal theme matches the watch and is acceptable.

## Workstream 3 — Watch history view

### User story

The user swipes from `LogView` one more step right to reach a new `HistoryView`. They see a 14-day bar chart (daily caffeine totals with a daily-limit reference line) above a scrollable list of days. SELECT on a day row opens that day's drink list.

### Architecture

**Navigation (extends existing swipe chain):**
```
Summary ↔ Timeline ↔ Log ↔ History
                            ↓ SELECT
                        DayDetailView (new)
```

All transitions use `switchToView` (existing pattern, no stack accumulation). `DayDetailView` is pushed with `WatchUi.pushView` and popped with BACK.

**New files:**
- `source/HistoryView.mc` — renders bar chart + day list. Extends `WatchUi.View`.
- `source/HistoryDelegate.mc` — handles UP/DOWN (scroll day list), SELECT (drill into day), swipe LEFT (back to LogView), BACK (exit widget). Extends `WatchUi.BehaviorDelegate`.
- `source/DayDetailView.mc` — reuses `LogView`-style rendering with data filtered to one specific day. Title row shows the date + daily total.
- `source/DayDetailDelegate.mc` — BACK pops, returning to `HistoryView`.

**Modified files:**
- `source/StorageManager.mc` — add daily-totals persistence and roll-up logic (see Data model).
- `source/HalfLifeCaffeineApp.mc` — call `storageManager.rollUpYesterday()` in `getInitialView()` and `onStop`.
- `source/LogDelegate.mc` — make RIGHT swipe go to `HistoryView` instead of being a no-op.

### Layout (round 240×240 to 454×454, scales proportionally)

- Top ~45% of height: 14-bar daily chart.
  - Y-axis: max of (daily limit × 1.1, highest bar × 1.1), so the daily-limit line is always visible.
  - Dashed red horizontal line at daily limit mg.
  - Bar colors: `Colors.ACCENT` (<80% limit), yellow (80–100%), red (>100%). Reuse existing palette, no new colors.
  - X-axis: oldest day on the left, today on the right. Today's bar updates live since it reads from today's doses + daily-totals store.
- Lower ~50%: 4–5 visible day rows (date `MMM d`, total `NNN mg`, tiny progress bar colored same as above).
  - Scrollable with UP/DOWN. Selected row highlighted with `Colors.ACCENT` border.
  - Day list is sorted newest-first. Top entry = today (matches rightmost bar).
- Horizontal inset: 10% on each side (round-screen safe; matches `TimelineView`/`LogView` pattern).
- All coordinates computed with integer math (SDK 9.1.0 drawing-coordinate gotcha per CLAUDE.md).

### Data model

Two storage tables:

1. **`doses`** (existing) — 14-day retention, individual entries. Unchanged. Powers `DayDetailView` for days within the 14-day window.
2. **`dailyTotals`** (new) — 90-day rolling window. Array of `[ymd:Number, totalMg:Number, doseCount:Number]`. Stored as a flat array of primitives (per the project convention that dicts don't round-trip reliably through Storage). `ymd` is an integer like `20260424`.

**`dailyTotals` rules:**
- One entry per day.
- Day's entry is written when the day rolls over (today becomes yesterday). Until then, "today" is computed on the fly from active `doses`.
- Pruned to 90 entries max (drop oldest first).

### Roll-up logic (`StorageManager.rollUpYesterday`)

```
lastRolledYmd = Storage.getValue("lastRolledYmd") or 0
yesterday = ymdOf(now - 1 day)
if lastRolledYmd >= yesterday: return

for each ymd from lastRolledYmd+1 (or yesterday if lastRolledYmd==0) through yesterday:
    doses_for_day = filter(doses, time falls within ymd)
    total_mg = sum of doses_for_day[mg]
    count = doses_for_day.size()
    if total_mg > 0:
        append [ymd, total_mg, count] to dailyTotals

if dailyTotals.size() > 90: trim from front to 90
Storage.setValue("lastRolledYmd", yesterday)
save dailyTotals
```

Called from `HalfLifeCaffeineApp.getInitialView()` (widget open) and `onStop` (belt-and-suspenders). Idempotent — safe to call multiple times in the same day. First call on a fresh install creates no entries (no prior doses).

### `DayDetailView` data source

- If day is within 14-day window: filter `doses` by day, show drink list (time, name, mg) — same rendering as `LogView`.
- If day is older than 14 days: show only the daily total from `dailyTotals` plus a subtle "Details expired after 14 days" footer. The daily total survives; individual drinks don't.

### Glance view

Unchanged. Glance process does not need `HistoryView` or daily-totals data; it renders only "now."

## Testing and release plan

### Simulator test matrix (5 device runs)

| Device | Validates |
|---|---|
| `vivoactive5` | Regression — current primary target |
| `fenix5plus` | Smallest 240×240 MIP + 64-color palette |
| `fenix7pro` | Mid-range reference |
| `fenix847mm` | Large AMOLED, newest SoC |
| `venusq2` | Rectangular screen; layout doesn't assume round |

Each run verifies: swipe chain `Summary → Timeline → Log → History → DayDetail → back`, bar chart renders, day list scrolls, drill-into-day works, glance unchanged, settings page loads without crash.

### On-device testing

Primary developer watch (vivoactive 5):
- End-to-end flow: log a drink, force roll-over, confirm history updates. Options for forcing roll-over during testing: advance the simulator's system clock past midnight, or add a temporary `(:test)`-gated helper that stubs "yesterday" relative to a fixed offset. Pick one during implementation.
- Companion page hosted on GitHub Pages, opened via real phone and Connect IQ Mobile: load defaults, edit preset, save, confirm preset reaches watch.

### Release checklist

1. Bump the app version to 1.1.0. The current `manifest.xml` does not carry a version attribute, so version is set through the Connect IQ store dashboard when uploading the `.iq`.
2. Update the store listing "What's New" with: new device support summary, companion settings page, watch history view.
3. Build release `.iq`:
   ```
   monkeyc -e -f monkey.jungle -y D:/Garmin/Key/developer_key -o bin/HalfLifeCaffeine.iq
   ```
   Confirm clean build across all 45 devices (17 + 28).
4. Upload `.iq` to Connect IQ app-store dashboard.
5. Set "Settings URL" to the GitHub Pages URL.
6. Submit for review.

### Rollback

- `.iq` artifacts kept in git tags per release.
- If post-release regression: revert manifest additions + HistoryView wiring on a hotfix branch, rebuild, resubmit as v1.1.1.
- Settings URL is metadata-only; can be reverted independently without rebuilding.
