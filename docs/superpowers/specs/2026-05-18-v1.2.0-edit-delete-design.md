# HalfLife Caffeine v1.2.0 ‚Äî Design (edit/delete logged drinks)

**Date:** 2026-05-18
**Target release:** v1.2.0
**Prior spec:** `2026-04-24-v1.1.0-design.md`

## Overview

Add the ability to **edit the time of** or **delete** a drink that's already been logged today, addressing two user pain points:

1. **Mis-tap.** User picks the wrong drink from the preset menu. Today's only recourse is to live with the wrong entry. v1.2.0 adds in-place delete; the user can then re-log correctly.
2. **Forgot to log promptly.** User logs a drink an hour after actually consuming it. The dose's `:time` is wrong, which corrupts decay calculations, the timeline graph, and the "minutes to safe" estimate. v1.2.0 adds preset time offsets (`-15 min` through `-3 hours`, plus `+15 min` / `+30 min` for fine-tuning).

Both actions happen on the existing `LogView` ("Today's Drinks"). LogView becomes interactive: UP/DOWN moves a cursor through entries; SELECT opens an action menu. This mirrors the cursor pattern already used in `HistoryView`.

A second concern surfaced during design: the companion page accumulates `drinkHistory` in localStorage and renders it in the History and Trends tabs. With watch-side edits/deletes, the phone mirror would drift. The fix is a small protocol extension ‚Äî watch sends a full-day resync on any change, companion replaces today's entries instead of blindly appending.

## Out of scope (deferred)

- Editing drinks on **past days** (via `DayDetailView`). The motivating cases are all same-day. Past-day edits would require also mutating the `dailyTotals` roll-up store and a UI flow inside `DayDetailView`; defer.
- Editing **dose amount (mg)** or **name**. Delete-and-re-add covers the "wrong preset" case. If demand emerges later, the `EditDrinkMenuDelegate` is the natural seam to extend.
- **Minute-precision time edits.** Preset offsets in coarse steps are sufficient for caffeine's 5.7h half-life. A custom stepper view is YAGNI for now.
- **Delete confirmation prompt.** Re-logging is cheap; the confirm step would be friction. Accidents are recoverable.
- **Stable dose IDs.** Identity-by-index is fine for today-only scope (‚â§ ~10 doses); full-day resync sidesteps the identity problem on the wire.

---

## Locked decisions

| Decision | Choice |
|---|---|
| Scope | Today-only, via LogView |
| What's editable | Time + Delete (no mg, no name) |
| Interaction | Cursor on LogView (mirror HistoryView) ‚Äî UP/DOWN moves; SELECT opens action menu |
| Time edit UX | `Menu2` of preset offsets: `-15 min`, `-30 min`, `-1 hour`, `-2 hours`, `-3 hours`, `+15 min`, `+30 min` |
| Delete confirmation | None ‚Äî instant |
| Phone sync | Full-day resync via new `mode: "replace-day"` field on the `drinks` message |
| Cross-midnight backward | Allowed (drink legitimately moves to yesterday) |
| Future time | Clamped at `nowEpoch` (a drink can't be in the future) |

## Architecture

Three layers, mirroring the existing manager + view + delegate pattern.

### Model (`CaffeineModel`)

Add two mutation methods, plus an index-mapping helper for safe view-to-model translation.

- **`removeDoseAt(doseIndex)`** ‚Äî splice from the master `_doses` array. Out-of-bounds index is a no-op (defensive ‚Äî if the cursor lands on a row right as another code path mutates the array, we don't want to crash).
- **`adjustDoseTime(doseIndex, offsetSeconds, nowEpoch)`** ‚Äî adds `offsetSeconds` to `_doses[doseIndex][:time]`. Clamps the result to `<= nowEpoch` (future-clamp rule). Negative offsets crossing midnight are accepted; the new `:time` may land on yesterday. Returns the resulting `:time` so the orchestrator can decide which days to resync.
- **`getTodayLogIndices(nowEpoch)`** ‚Äî returns an array of `_doses` indices in the same order `getTodayLog` yields the dose dicts (chronological-ascending, matching `addDose` order). Avoids splice-by-identity, which is fragile because dose dicts are not unique (two doses at the exact same `:time` and `:mg` would be indistinguishable).

Indexing convention: the model returns chronological-ascending order; LogView renders newest-first. The view reverses the model's output when building its `dayIndices` cache, so `dayIndices[0]` is the newest dose's `_doses` index. All view-facing references (`selectedIndex`, `getSelectedDoseIndex`) use the rendered (newest-first) order; the model's mutation methods (`removeDoseAt`, `adjustDoseTime`) take the `_doses` index directly.

### View (`LogView`)

Adopt the HistoryView cursor pattern. Three new instance fields:

- `selectedIndex` ‚Äî index into the **rendered (newest-first) list**. Default `0` = the most recently logged drink, the typical mis-tap or forgot-to-log target.
- `dayIndices` ‚Äî cached array, rebuilt in `onUpdate`, mapping rendered index ‚Üí `_doses` index.
- `dayDoses` ‚Äî cached today-log array, same length as `dayIndices`, also rebuilt in `onUpdate`.

Three new methods:

- `moveSelection(delta)` ‚Äî clamps `selectedIndex` to `[0, dayDoses.size() - 1]`, requests redraw if it changed.
- `getSelectedDoseIndex()` ‚Äî returns `dayIndices[selectedIndex]`, or `-1` if the log is empty.
- The existing onUpdate gains a selection-highlight pass: before drawing the line for `selectedIndex`, fill a `Colors.TRACK` rectangle behind it. Identical visual to `HistoryView.drawDayList`.

Edge: after a delete or a cross-midnight time-edit, the displayed list shrinks. `moveSelection(0)` is invoked by the orchestrator after mutations to re-clamp `selectedIndex` to the new last valid index. (Implementation: just call `moveSelection(0)`; the clamp logic inside handles "selectedIndex is now beyond end".)

### Delegate (`LogDelegate`)

Existing: `onPreviousPage` (swipe up ‚Üí TimelineView), `onNextPage` (swipe down ‚Üí HistoryView), `onKey` for non-touch UP/DOWN dispatch to those.

Changes:

- **Re-purpose `onKey` for UP/DOWN to move the cursor**, not navigate pages. Swipes (touch) and SWIPE_UP / SWIPE_DOWN delegate calls continue to use `onPreviousPage`/`onNextPage`.
- Add `onSelect` ‚Äî pushes a `Menu2` titled `"Edit Drink"` with two items: `Edit time` and `Delete`. Pass the `LogView` reference and the selected dose index into a new `EditDrinkMenuDelegate`.

This is a behavioral change for non-touch devices: UP/DOWN no longer page-switches on LogView. Page-switching on those devices still works via swipe gestures and via the action-menu route (SELECT ‚Üí back is the natural "I'm done editing, get me out" path). Acceptable trade-off ‚Äî without UP/DOWN paging, the cursor is impossible to drive on non-touch hardware.

The delegate holds a reference to its `LogView` instance, same as `HistoryDelegate` does.

### Two new files

- **`source/EditDrinkMenuDelegate.mc`** ‚Äî `WatchUi.Menu2InputDelegate` subclass. Holds the dose index. On `onSelect(item)`:
  - `Edit time` ‚Üí push another `Menu2` of offsets, using `EditTimeMenuDelegate` (passes the dose index along).
  - `Delete` ‚Üí `app.removeDose(doseIndex)`, then `WatchUi.popView(SLIDE_RIGHT)` back to LogView.
- **`source/EditTimeMenuDelegate.mc`** ‚Äî `WatchUi.Menu2InputDelegate` subclass. Holds the dose index. On `onSelect(item)`: reads the offset from the item id (negative or positive seconds), calls `app.adjustDoseTime(doseIndex, offsetSeconds)`, then `popView` twice (pops the time-offsets menu and the Edit-Drink menu, returning to LogView).

The two menus are deliberately constructed inline in the delegates rather than via XML resources ‚Äî they're trivial and not localized. Matches `DrinkMenuDelegate`'s approach.

### Orchestrator (`HalfLifeCaffeineApp`)

Two new methods alongside the existing `logDrink`:

- **`removeDose(doseIndex)`** ‚Äî `caffeineModel.removeDoseAt(doseIndex)`, save doses to storage, sync today to phone (see Sync section), `WatchUi.requestUpdate()`.
- **`adjustDoseTime(doseIndex, offsetSeconds)`** ‚Äî capture `oldYmd` from the dose's pre-edit time, call `caffeineModel.adjustDoseTime(doseIndex, offsetSeconds, now)`, capture `newYmd` from the returned time. Prune expired (`pruneExpiredDoses(now)`). Save doses. Sync today's day to phone ‚Äî and if `newYmd != oldYmd` (cross-midnight case), also sync yesterday. `WatchUi.requestUpdate()`.

Neither method re-runs `alertManager.checkAlerts` ‚Äî the daily total didn't increase (delete reduces it; time-edit keeps it the same). Re-running the limit alerts could only down-fire them, which the once-per-day tracking in AlertManager doesn't reverse anyway.

## Sync protocol extension

The current watch ‚Üí phone message is:

```json
{"type": "drinks", "data": [{"mg": 95, "time": 1747573200, "name": "Drip Coffee (S)"}, ...]}
```

The companion's `onWatchData` (in `companion/settings/index.html`) appends `data.data` to `drinkHistory` in localStorage. There is no dedup, no replace, no concept of "this is the authoritative list for day X."

### Change

Add a `mode` field to the watch ‚Üí phone message:

```json
{
  "type": "drinks",
  "mode": "replace-day",
  "ymd": 20260518,
  "data": [{"mg": 95, "time": 1747573200, "name": "Drip Coffee (S)"}, ...]
}
```

When `mode === "replace-day"`:
- The `ymd` field identifies the day (local YYYYMMDD int, same convention as `Util.ymdFromEpoch`).
- `data` is the **complete** dose list for that day, not an incremental delta.
- Companion is expected to drop existing `drinkHistory` entries that fall on that day, then append `data`.

When `mode` is absent (older watch versions) or `"append"`:
- Existing behavior. `drinkHistory.push(...data)`. No filtering.

This is backwards-compatible: a v1.1.0 watch keeps sending append messages and the companion keeps appending. A v1.2.0 watch always sends replace-day messages and the companion replaces.

### Watch-side changes

`SyncManager`:

- New method `syncDayToPhone(doses, ymd)` ‚Äî filters `doses` to entries whose `Util.ymdFromEpoch(:time) == ymd`, sends `{type: "drinks", mode: "replace-day", ymd, data: [...]}` via `Communications.transmit`. Does **not** consult `lastSyncTime`.
- Existing `syncToPhone` (incremental, gated by `lastSyncTime`) is **removed**. All sync goes through `syncDayToPhone` now.
- `lastSyncTime` storage key and accessors (`getLastSyncTime`/`saveLastSyncTime` on `StorageManager`) become dead code. Leave them for one release as a safety net; remove in v1.3.0. (They cost ~30 LOC and a couple of bytes of storage.)

`HalfLifeCaffeineApp.logDrink` switches from `syncManager.syncToPhone(...)` to `syncManager.syncDayToPhone(caffeineModel.getDoses(), Util.ymdFromEpoch(now))`. This means every drink-add now triggers a full-day resync rather than an incremental delta. Cost: at ~10 doses/day worst case, the payload is small (under ~500 bytes encoded). Benefit: the companion always converges, even if the watch missed some prior sync.

### Companion-side change

In `companion/settings/index.html`'s `onWatchData`:

```js
function onWatchData(data) {
    if (!data || data.type !== 'drinks' || !data.data) { return; }
    if (data.mode === 'replace-day' && typeof data.ymd === 'number') {
        // Drop existing entries for the given ymd, then append the new full-day list.
        drinkHistory = drinkHistory.filter(function(d) {
            return localYmd(d.time) !== data.ymd;
        });
        for (var i = 0; i < data.data.length; i++) {
            drinkHistory.push(data.data[i]);
        }
    } else {
        // Legacy / append mode ‚Äî preserve existing behavior.
        for (var i = 0; i < data.data.length; i++) {
            drinkHistory.push(data.data[i]);
        }
    }
    localStorage.setItem('drinkHistory', JSON.stringify(drinkHistory));
}

function localYmd(epochSeconds) {
    var d = new Date(epochSeconds * 1000);
    return d.getFullYear() * 10000 + (d.getMonth() + 1) * 100 + d.getDate();
}
```

The companion file lives in two places: `companion/settings/index.html` on `main` (canonical source) and `settings/index.html` on the `gh-pages` branch (production). Both must be updated as part of this release. The promotion workflow established in v1.1.0 (`git show main:companion/settings/index.html > settings/index.html` on `gh-pages`) applies unchanged.

### Cross-midnight sync handling

`HalfLifeCaffeineApp.adjustDoseTime` captures `oldYmd` (from the dose's pre-edit `:time`) and `newYmd` (from the post-edit `:time` returned by the model). It then issues `syncDayToPhone` for `oldYmd`, and if `newYmd != oldYmd`, a second call for `newYmd`. Each call sends the **post-mutation** full-day dose list for its day:

- Sync `oldYmd` (today) ‚Äî full list no longer includes the moved dose; companion's mirror loses it from today.
- Sync `newYmd` (yesterday) ‚Äî full list now includes the moved dose; companion's mirror gains it on yesterday.

In the common no-midnight-cross case, `newYmd == oldYmd` and only one call runs.

## Edge cases & UX details

- **Empty log.** If `getTodayLog` is empty, LogView renders "No drinks today" (existing behavior). `onSelect` and `onKey` early-return without effect; `getSelectedDoseIndex()` returns `-1`; `EditDrinkMenuDelegate` is never instantiated.
- **Selection after mutation.** After delete or cross-midnight edit, `selectedIndex` may exceed the new list size. `moveSelection(0)` (called by the orchestrator post-mutation, before `requestUpdate`) re-clamps to `[0, dayDoses.size() - 1]`.
- **Single-entry log + delete.** Edge of the above: 1 entry, user deletes it. List is now empty, `selectedIndex` clamps to `0` but `dayDoses.size() == 0`. Subsequent `onSelect` early-returns on `getSelectedDoseIndex() == -1`. LogView re-renders "No drinks today."
- **Offset menu rendering.** Menu items are static strings with no subLabel:
  - `-15 min`, `-30 min`, `-1 hour`, `-2 hours`, `-3 hours`
  - `+15 min`, `+30 min`
  Item id encodes the offset in seconds (e.g. `-1800` for `-30 min`). The delegate reads the id directly.
- **Future-clamp behaviour.** If user is at 14:00 and a drink was logged at 13:55 and they pick `+30 min`, the math is `13:55 + 30:00 = 14:25` which is `> 14:00`, so clamp to `14:00`. No visible error; the drink lands at `now`. (Documented behaviour, not a surprise ‚Äî `+15`/`+30` are intended for small forward nudges, and clamp matches user mental model.)
- **Glance is unaffected.** Edits/deletes change `_doses`; the glance reads `_doses` fresh on every render, so it reflects changes on the next glance refresh. No additional wiring needed.
- **Storage persistence.** Both `removeDose` and `adjustDoseTime` call `storageManager.saveDoses(...)` synchronously before returning. The widget's lifecycle already saves on `onStop` as a backup, so even if `requestUpdate` happens before save (it shouldn't ‚Äî they run in order in the same method), the change persists.
- **Roll-up interaction.** A cross-midnight backward edit moves a dose from today to yesterday. If `lastRolledYmd` already covers yesterday, the roll-up store is stale for that day. Two cases:
  1. Yesterday hasn't been rolled yet (lastRolledYmd < yesterdayYmd) ‚Äî next `rollUpYesterday` call (on widget open or onStop) sees the dose and includes it correctly.
  2. Yesterday has already been rolled (lastRolledYmd == yesterdayYmd) ‚Äî the dose silently fails to update the daily total for yesterday. HistoryView's bar for yesterday is stale.

  For v1.2.0 this is acceptable ‚Äî it's an edge case (editing a drink across midnight) and the discrepancy is small (one dose's mg). Documenting it here so future work knows to consider: a more correct fix is to invalidate `lastRolledYmd` and re-roll when a cross-midnight edit happens. Defer.

## Testing

### Unit tests (`test/CaffeineModelTest.mc` additions)

- `testRemoveDoseAtRemovesCorrectEntry` ‚Äî three doses, remove middle (index 1), assert remaining doses are the first and third.
- `testRemoveDoseAtIsNoopOnOutOfBoundsIndex` ‚Äî call with `index = size`, assert size unchanged; call with `index = -1`, assert size unchanged.
- `testAdjustDoseTimeAppliesOffset` ‚Äî dose at known time, offset by `-3600`, assert `:time` decreased by 3600.
- `testAdjustDoseTimeClampsPositiveOffsetToNow` ‚Äî dose 5 minutes before `now`, offset by `+1800` (30 min), assert resulting `:time == now`.
- `testAdjustDoseTimeAllowsNegativeOffsetCrossingMidnight` ‚Äî dose just after midnight (`now = midnight + 60`), offset by `-7200` (2 hours), assert resulting `:time == midnight - 7140` (no clamp).
- `testGetTodayLogIndicesAlignsWithGetTodayLog` ‚Äî set up `_doses` with three of today's doses and two from yesterday, assert `getTodayLogIndices(now)` is `[i0, i1, i2]` where `_doses[i0..i2]` are the today entries (in stored order).

### Simulator smoke tests

Run the full cursor + edit flow on three reference devices:

- **`vivoactive5`** (round, touch + buttons) ‚Äî baseline. Verify touch swipes still page through Summary ‚Üí Timeline ‚Üí Log ‚Üí History. On Log, verify UP/DOWN move the cursor and SELECT opens the action menu. Edit and delete a drink; verify LogView reflects changes immediately.
- **`fenix7pro`** (round, button-only) ‚Äî verify non-touch UP/DOWN moves the cursor (not page-switches). Verify BACK exits the widget cleanly (same as HistoryView on non-touch today ‚Äî see "Non-touch navigation trade-off" below).
- **`venusq2`** (rectangular, touch) ‚Äî verify cursor highlight rectangle doesn't overflow the screen. Verify Menu2 rendering on rectangular screens works (it should ‚Äî `DrinkMenuDelegate`'s preset menu already proves this in v1.1.0).

**Non-touch navigation trade-off.** On non-touch devices (fenix7, fenix7pro, fenix8 family, fr955/965), UP/DOWN buttons are the only swipe-equivalent input. v1.1.0's `HistoryView` already binds UP/DOWN to its cursor and accepts that non-touch users can only leave HistoryView via BACK (which exits the widget). v1.2.0 introduces the same trade-off on LogView. The user reaches LogView from TimelineView (via UP/DOWN there) and can exit via BACK; they cannot page-switch directly to HistoryView from LogView on non-touch. Acceptable: the cursor utility is the primary value of the screen, and re-entering the widget to reach HistoryView is cheap.

### Phone round-trip test

After implementation, manually verify:

1. Add a drink on the watch ‚Üí companion's History tab shows it.
2. Edit the drink's time ‚Üí companion's History tab reflects the new time (no duplicate, no stale entry).
3. Delete the drink ‚Üí companion's History tab no longer shows it.
4. Cross-midnight backward edit ‚Üí companion's History tab shows the drink under yesterday's date (today's entry is removed).

This requires the v1.2.0 companion to be deployed (or accessible via file:// with mocked `onWatchData` calls). Companion can be tested offline by constructing the message JSON in browser DevTools and calling `onWatchData(msg)` directly.

## Implementation order (rough)

Detailed task breakdown is the implementation plan's job, but for sanity:

1. Model layer first ‚Äî `removeDoseAt`, `adjustDoseTime`, `getTodayLogIndices` with unit tests. Pure, no dependencies on view.
2. Sync layer ‚Äî `syncDayToPhone`, retire `syncToPhone`. Update `logDrink` to use the new method.
3. Companion ‚Äî update `companion/settings/index.html` (and later `gh-pages`) with the replace-day branch.
4. View + delegate ‚Äî `LogView` cursor, `LogDelegate.onKey`/`onSelect`, the two new menu delegates.
5. Orchestrator ‚Äî `removeDose` and `adjustDoseTime` on `HalfLifeCaffeineApp`. Wire to delegates.
6. Cross-device simulator pass.
7. On-device phone round-trip.
8. Release.

## Risks

- **Non-touch UP/DOWN repurposing.** Re-binding UP/DOWN on LogView from page-nav to cursor-nav is the largest behavioral change. Users of non-touch devices will lose UP/DOWN paging on this specific screen. The action menu and BACK key still provide a navigation path, but it's a small step backward in UX for non-touch users. Mitigated by: it's only LogView (other screens still page with UP/DOWN), and BACK reliably exits.
- **Sync protocol drift.** Adding `mode` to an existing message type is backwards-compatible only if the companion deployed at any given time is current. Mismatched watch + companion versions (new watch + old companion) will revert to append behavior ‚Äî edits silently fail to propagate, no crash, but the companion's data is stale until the next companion deploy. Acceptable for this release; document in the changelog.
- **`lastSyncTime` dead code.** Leaving it in place for v1.2.0 is a small carrying cost. If someone touches `SyncManager` between v1.2.0 and v1.3.0, the dead path could mislead them. Removal in v1.3.0 is a TODO worth tracking in a follow-up note.
- **Cross-midnight roll-up staleness.** Documented above. Real but edge-case. Defer the fix.
